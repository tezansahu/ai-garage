# Storytelling Frameworks Reference

Each framework below includes a description, best-fit context, and a rough structural outline.
When presenting to the user, generate a **tailored rough outline** for their specific project using each framework — don't show this template verbatim.

---

## 1. Problem → Solution → Impact (PSI)
**Best for:** Technical demos, product pitches, and investor presentations where clarity > flair.
**Tone:** Direct, logical, confident.

**Structure:**
1. Frame the problem with data (who suffers, how much)
2. Expose gap in existing solutions (why current answers fall short)
3. Introduce product as the precise fix
4. Demo / proof
5. Show impact: what changes for the user, the company, the industry

**Strength:** Universal. Judges understand it instantly. Easy to time.
**Weakness:** Can feel formulaic if executed without emotion.

---

## 2. Before → After → Bridge (BAB)
**Best for:** Consumer-facing or developer-experience products. High emotional contrast.
**Tone:** Vivid, empathetic, transformational.

**Structure:**
1. Paint "Before" state in visceral detail — frustration, confusion, waste
2. Paint "After" state — the world where the problem is solved (aspirational)
3. Bridge: "Here's what makes this possible" → introduce product
4. Demo as proof the bridge exists
5. Invitation: this future is available now

**Strength:** Emotionally memorable. The contrast is the hook.
**Weakness:** Requires strong "Before" writing. Weak if problem is niche.

---

## 3. The Pixar Story Spine
**Best for:** Products solving a relatable daily pain. Great for non-technical audiences.
**Tone:** Narrative, human, story-driven.

**Structure:**
1. "Once upon a time..." → set the scene, introduce the protagonist (the developer, the team)
2. "Every day..." → describe their normal, inefficient routine
3. "Until one day..." → the breaking point / inciting incident
4. "Because of that..." → consequences compound
5. "Until finally..." → product arrives, problem resolves
6. "Ever since then..." → the new normal (impact)

**Strength:** Deeply memorable. Judges feel the story, not just the facts.
**Weakness:** Harder to fit technical architecture naturally. Requires confident delivery.

---

## 4. Situation → Complication → Resolution (SCR)
**Best for:** Enterprise, Microsoft-internal, executive audiences. Clean and strategic.
**Tone:** McKinsey-style. Precise, structured, authoritative.

**Structure:**
1. Situation: Establish shared context (what is already true and agreed upon)
2. Complication: Introduce the tension that disrupts the situation (what's broken/missing)
3. Resolution: Your product as the answer — with evidence

Then add:
4. How it works (architecture / mechanism)
5. Strategic impact (business value, scalability)

**Strength:** Signals sophistication. Resonates with technical decision-makers.
**Weakness:** Can feel cold if no human element. Needs strong data at each stage.

---

## 5. The Hero's Journey (Condensed)
**Best for:** Products that empower the user. Strong brand storytelling.
**Tone:** Cinematic, energizing, aspirational.

**Structure:**
1. The Hero (developer) in their ordinary world — struggling with chaos
2. The Call to Adventure — the problem gets unavoidable
3. Meeting the Mentor / Getting the Tool — product is introduced
4. The Transformation — demo shows the hero now capable of what they couldn't do before
5. Return with the Elixir — new state of the world, impact on the industry

**Strength:** Makes the user the protagonist. Extremely memorable.
**Weakness:** Easiest to over-dramatize. Needs confident, punchy writing to land.

---

## Hybrid / Combination Notes

For pitches with both technical evaluators AND business stakeholders, a common winning pattern is:
- **Outer shell:** BAB or PSI (emotional hook + clear logic)
- **Architecture section:** SCR micro-structure (situation=existing stack, complication=gap, resolution=your design)
- **Impact:** Hero's Journey close (the developer as transformed protagonist)

When the user selects a framework, adapt this hybrid insight if it strengthens their specific project.
